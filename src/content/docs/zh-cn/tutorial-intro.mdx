**Flake FHS** 通过引入一套标准的**文件系统层次结构 (FHS)**，将 Flake Outputs 直接映射到文件系统。**只要看到了目录结构，你就知道对应的 Output 在哪里。**

## 目标用户

Flake FHS 特别适合你，如果：

- **你是 Nix 新手**，想迁移到 Flake 但不知道从何入手
- **你有丰富的 NixOS 使用经验**，随着配置仓库膨胀，有强烈的意愿重构以提升可维护性
- **你只想要开箱即用的 Flake 体验**，但并不想关注 Flake 的设计细节
- **你想学习 Flake 项目组织的最佳实践**，即使最终不使用本框架

<Aside type="note">
如果你已经对 Flake 有深入理解，并且对当前的项目结构完全满意，Flake FHS 可能不会带来显著的额外价值。
</Aside>

## 为什么选择 Flake FHS

传统 Nix Flake 开发存在以下核心问题：

### 问题一：样板代码重复

添加一个软件包，传统方式需要这样的 `flake.nix`：

```nix
{
  inputs.nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";

  outputs = { self, nixpkgs }:
    let
      systems = [ "x86_64-linux" "aarch64-linux" ];
      forAllSystems = f: nixpkgs.lib.genAttrs systems (system: f {
        pkgs = import nixpkgs { inherit system; };
      });
    in
    {
      packages = forAllSystems ({ pkgs }: {
        hello = pkgs.writeShellScriptBin "hello" ''
          echo "Hello from Flake FHS!"
        '';
      });
    };
}
```

**问题**：
- `forAllSystems` 的模式在每个项目中重复
- 系统遍历逻辑与包定义混在一起
- 添加新包需要修改 `flake.nix`，容易出错

**Flake FHS 解决方案**：

`flake.nix` 只需要这样：

```nix
{
  inputs.nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
  inputs.flake-fhs.url = "github:luochen1990/flake-fhs";

  outputs = inputs@{ flake-fhs, ... }:
    flake-fhs.lib.mkFlake { inherit inputs; } { };
}
```

添加包时，只需创建 `pkgs/hello.nix`：

```nix
{ writeShellScriptBin }:
writeShellScriptBin "hello" ''
  echo "Hello from Flake FHS!"
''
```

框架会自动遍历系统、发现包定义、注入依赖、生成 outputs。

### 问题二：源码定位困难

接手一个新项目时，想找到某个包的源码位置，往往需要：
- 阅读复杂的 `flake.nix` 文件
- 理解项目特有的组织方式
- 每个项目都有自己的目录结构约定

**Flake FHS 解决方案**：

定义清晰的目录到 flake outputs 的映射关系：

| 目录 | 对应 Output | Nix 命令 |
|------|-------------|-----------|
| `pkgs/` | `packages.<system>.<name>` | `nix build .#<name>` |
| `apps/` | `apps.<system>.<name>` | `nix run .#<name>` |
| `hosts/` | `nixosConfigurations.<name>` | `nixos-rebuild switch --flake .#<name>` |
| `modules/` | `nixosModules.<name>` | - |
| `checks/` | `checks.<system>.<name>` | `nix flake check .#<name>` |
| `shells/` | `devShells.<system>.<name>` | `nix develop .#<name>` |
| `lib/` | `lib.<name>` | `nix eval .#lib.<name>` |
| `templates/` | `templates.<name>` | `nix flake init --template .#<name>` |

看到目录结构就知道对应的 output 在哪里，代码位置可预测。

### 问题三：模块拆分繁琐

传统的 NixOS 模块拆分需要：

```nix
{
  imports = [
    ./modules/tailscale.nix
    ./modules/ssh.nix
    ./modules/firewall.nix
  ];
}
```

**问题**：
- 需要手动维护导入列表
- 容易遗漏某个模块
- 细粒度拆分时每个文件都要写 `lib.mkIf` 样板代码

**Flake FHS 解决方案**：

将模块放入 `modules/` 目录，自动发现并注册。结合 **Guarded Module** 特性，还可以自动生成 `enable` 选项并包装条件逻辑。

### 问题四：子作用域包管理困难

nixpkgs 的 `pkgs/by-name/` 机制只能处理顶层 `pkgs` 作用域的包。对于子作用域（如 `python3Packages`、`haskellPackages`），需要：
- 手动调用 `pkgs.python3Packages.callPackage`
- 在每个包中重复指定作用域
- 依赖注入逻辑分散，难以统一管理

**Flake FHS 解决方案：Scoped Package Tree**

通过 `scope.nix` 定义子作用域，子目录中的包自动继承：

```
pkgs/
└── python/
    ├── scope.nix           # 定义作用域
    ├── my-app/
    │   └── package.nix     # 自动继承 python3Packages 作用域
    └── my-lib/
        └── package.nix
```

### 问题五：缺乏统一规范

不同的 Nix 项目往往采用不同的目录结构、命名约定、组织方式，导致：
- 团队协作时需要学习新项目的约定
- 难以开发通用的自动化工具
- 代码审查和理解的成本增加

**Flake FHS 解决方案**：

提供标准化的目录结构，适用于所有项目：

```
flake.nix              # 框架配置
pkgs/                  # 软件包
apps/                  # 应用程序
hosts/                 # NixOS 配置
modules/               # NixOS 模块
shells/                # 开发环境
```

## 核心场景对比

理解了上述问题和解决方案后，让我们通过具体场景直观感受 Flake FHS 的优势。

### 场景一：模块拆分

将配置拆分到多个模块，实现关注点分离。

<Tabs>
<TabItem label="传统方式 😫">

**简单拆分**：需要手动维护 `imports` 列表：

```nix
// configuration.nix
{
  imports = [
    ./modules/desktop.nix
    ./modules/network.nix
    ./modules/services.nix
    // 容易忘记添加新模块
  ];
}
```

**细粒度拆分**：每个文件都要写样板代码：

```nix
// modules/my-website/options.nix
{ lib, ... }:
{
  options.my-website.enable = lib.mkEnableOption "my website";
}
```

```nix
// modules/my-website/nginx.nix
{ config, lib, ... }:
{
  config = lib.mkIf config.my-website.enable {
    services.nginx.enable = true;
  };
}
```

```nix
// modules/my-website/redis.nix
{ config, lib, ... }:
{
  config = lib.mkIf config.my-website.enable {
    services.redis.enable = true;
  };
}
```

```nix
// configuration.nix
{
  imports = [
    ./modules/my-website/options.nix
    ./modules/my-website/nginx.nix
    ./modules/my-website/redis.nix
    // 容易忘记添加新文件
  ];
}
```

**拆分有成本**：需要手动定义 options、维护 imports、重复写样板代码。

</TabItem>

<TabItem label="Flake FHS ✨">

**简单拆分**：只需创建目录，自动发现：

```
modules/
├── desktop/
│   ├── options.nix
│   └── config.nix
├── network/
│   ├── options.nix
│   └── config.nix
└── services/
    ├── options.nix
    └── config.nix
```

**细粒度拆分**：Guarded Module 自动包装，无需样板代码：

```
modules/
└── my-website/
    ├── options.nix    # 可以留空，enable option 自动生成
    ├── nginx.nix      # 自动包装 mkIf
    └── redis.nix      # 自动包装 mkIf
```

```nix
// modules/my-website/nginx.nix
{ ... }:
{
  // 自动被包装成: lib.mkIf config.my-website.enable { ... }
  services.nginx.enable = true;
}
```

**拆分零成本**：随意拆分，无需样板代码。

</TabItem>
</Tabs>

### 场景二：多主机管理

在一个仓库中管理多台 NixOS 机器（laptop, server, vm 等）。

<Tabs>
<TabItem label="传统方式 😫">

每个主机一个 Flake 仓库，配置无法共享：

```
~/config-laptop/
├── flake.nix
└── configuration.nix

~/config-server/
├── flake.nix
└── configuration.nix
```

需要手动复制粘贴共享配置，难以保持同步。

</TabItem>

<TabItem label="Flake FHS ✨">

一个仓库管理所有主机，共享模块：

```
my-nixos/
├── flake.nix
├── modules/
│   ├── tweaks.nix       # 系统优化
│   └── my-website.nix   # 网站服务
└── hosts/
    ├── laptop/
    │   └── configuration.nix
    └── server/
        └── configuration.nix
```

```nix
// hosts/laptop/configuration.nix
{ ... }:
{
  networking.hostName = "laptop";
  tweaks.enable = true;       // 只启用系统优化
}
```

```nix
// hosts/server/configuration.nix
{ ... }:
{
  networking.hostName = "server";
  tweaks.enable = true;       // 共享模块
  my-website.enable = true;   // 主机特有
}
```

</TabItem>
</Tabs>

## 对比总结

| 维度 | 传统方式 | Flake FHS |
|------|----------|-----------|
| 样板代码 | 重复编写 `forAllSystems` 等 | 零样板代码 |
| 源码定位 | 需要阅读 `flake.nix` | 目录即映射，一目了然 |
| 模块拆分 | 手动维护 imports + 每个文件写 `mkIf` | 自动发现 + Guarded Module |
| 子作用域包 | 手动调用 `python3Packages.callPackage` | Scoped Package Tree |
| 项目规范 | 每个项目都有自己的约定 | 统一的目录标准 |
| 新增内容 | 修改 `flake.nix` | 创建文件即可 |
| 学习曲线 | 需要深入理解 Flake API | 只需了解目录约定 |
