## 问题背景

许多开发者担心将现有的大型 Nix Flake 项目迁移到 Flake FHS 需要一次性大规模重构，这可能导致：

- 高风险：一次性改动过多，容易引入错误
- 工作量大：需要暂停正常开发进行迁移
- 团队阻力：变化太大，团队适应困难

## Flake FHS 的解决方案

Flake FHS 设计之初就考虑了渐进式采用的需求，支持三种采用模式：

### 模式一：完全采用

所有 outputs 由 Flake FHS 自动生成：

```nix
{
  inputs.nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
  inputs.flake-fhs.url = "github:luochen1990/flake-fhs";

  outputs = inputs@{ flake-fhs, ... }:
    flake-fhs.lib.mkFlake { inherit inputs; } {
      # 所有 outputs 自动生成
    };
}
```

适用于：
- 新项目
- 愿意完全重构的现有项目

### 模式二：混合模式

部分 outputs 由框架生成，部分手动定义：

```nix
{
  inputs.nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
  inputs.flake-fhs.url = "github:luochen1990/flake-fhs";

  outputs = inputs@{ flake-fhs, ... }:
    flake-fhs.lib.mkFlake { inherit inputs; } {
      # 自动生成的 outputs
    }
    // {
      # 手动定义的额外 outputs
      overlays.default = final: prev: {
        customPackage = final.callPackage ./overlays/custom.nix { };
      };
      
      nixosModules.legacy = import ./legacy-module.nix;
    };
}
```

适用于：
- 正在迁移中的项目
- 有特殊需求无法用框架表达的 outputs

### 模式三：逐步迁移

按照优先级分阶段迁移：

#### 阶段 1：迁移包定义

```bash
# 1. 创建目录结构
mkdir -p pkgs

# 2. 将包定义移动到 pkgs/
# 从 flake.nix 中提取包定义到 pkgs/hello.nix
mv legacy-packages/hello.nix pkgs/hello.nix
```

此时可以：
- 先迁移最简单的包
- 保持其他 outputs 不变
- 验证包构建正常

#### 阶段 2：迁移模块

```bash
# 1. 创建模块目录
mkdir -p modules

# 2. 迁移 NixOS 模块
mv legacy-modules/tailscale.nix modules/tailscale/options.nix
touch modules/tailscale/config.nix  # 添加配置实现
```

此时可以：
- 模块仍然可以独立使用
- 逐步添加自动发现和启用选项

#### 阶段 3：迁移主机配置

```bash
# 1. 创建主机目录
mkdir -p hosts

# 2. 迁移 NixOS 配置
mv legacy-hosts/laptop hosts/laptop/configuration.nix
```

最后完成完整迁移。

## 迁移策略建议

### 优先级排序

按照以下优先级迁移：

1. **高优先级**：频繁修改的内容
   - 开发环境 (shells/)
   - 测试检查 (checks/)
   - 常用包 (pkgs/)

2. **中优先级**：相对稳定的内容
   - 应用程序 (apps/)
   - NixOS 模块 (modules/)

3. **低优先级**：很少变动的配置
   - NixOS 主机 (hosts/)
   - 库函数 (lib/)

### 并行工作策略

团队可以采用分支策略并行迁移：

```bash
# 每个人负责迁移一个目录
git checkout -b migrate/pkgs
git checkout -b migrate/modules
git checkout -b migrate/shells

# 完成后合并
git merge migrate/pkgs
git merge migrate/modules
git merge migrate/shells
```

### 验证策略

每个阶段都要验证：

```bash
# 阶段 1 验证
nix build .#hello  # 验证包构建
nix develop        # 验证开发环境

# 阶段 2 验证
nixos-rebuild build --flake .#laptop  # 验证配置

# 阶段 3 验证
nix flake check    # 全面验证
```

## 实际案例

### 案例：从传统 Flake 迁移

**迁移前**（传统 flake.nix）:

```nix
{
  outputs = { self, nixpkgs }:
    let
      system = "x86_64-linux";
      pkgs = import nixpkgs { inherit system; };
    in
    {
      packages.${system}.hello = pkgs.writeShellScriptBin "hello" ''
        echo "Hello"
      '';
      
      devShells.${system}.default = pkgs.mkShell {
        packages = [ pkgs.git pkgs.nixpkgs-fmt ];
      };
    };
}
```

**迁移后**（逐步迁移）:

```nix
{
  inputs.flake-fhs.url = "github:luochen1990/flake-fhs";

  outputs = inputs@{ flake-fhs, nixpkgs, ... }:
    flake-fhs.lib.mkFlake { inherit inputs; } {
      # 框架自动发现 pkgs/ 和 shells/
    };
}
```

```nix
# pkgs/hello.nix
{ writeShellScriptBin }:
writeShellScriptBin "hello" ''
  echo "Hello"
''
```

```nix
# shells/default.nix
{ mkShell, git, nixpkgs-fmt }:
mkShell {
  packages = [ git nixpkgs-fmt ];
}
```

## 优势总结

| 特性 | 传统迁移 | 渐进式采用 |
|------|----------|-----------|
| 风险 | 高（一次性大改） | 低（分步骤验证） |
| 停机时间 | 长期阻塞 | 无需停机 |
| 团队适应 | 需要培训后才能使用 | 可以边迁移边学习 |
| 回滚难度 | 困难（改动太大） | 容易（每次改动小） |
| 价值体现 | 迁移完成才能看到 | 每阶段都能获益 |
