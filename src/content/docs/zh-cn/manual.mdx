**为什么需要 Flake FHS？**

在维护多个 Nix Flake 项目时，我们经常面临以下问题：

1.  **样板代码重复**：每个项目都需要编写大量雷同的 `flake.nix` 代码来处理 inputs、systems 遍历和模块导入。
2.  **结构差异巨大**：缺乏统一的目录规范，导致接手不同项目时需要花费额外精力理解其文件组织方式。
3.  **工具链集成难**：由于缺乏标准化的目录语义，难以开发通用的自动化工具来辅助开发。

**Flake FHS** 通过引入一套标准的文件系统层次结构 (FHS)，将 Flake Outputs 直接映射到文件系统。**只要看到了目录结构，你就知道对应的 Output 在哪里。**

## 1分钟入门 (Tutorial)

<Steps>

1.  **初始化项目**

    Flake FHS 提供了多种初始化模板，以适应不同的项目需求：

    <Tabs>
      <TabItem label="全新 Nix 项目" icon="rocket">
        适用于作为 **nixos-config** 或**纯 Nix 软件包仓库**。

        <Tabs>
          <TabItem label="推荐 (短名)" icon="star">
            **特点**：目录名简洁 (`pkgs`, `hosts`)，符合社区习惯。

            ```bash
            nix flake init --template github:luochen1990/flake-fhs#short
            ```
          </TabItem>
          <TabItem label="标准 (长名)" icon="document">
            **特点**：目录名完整 (`packages`, `nixosConfigurations`)，严格对应 Nix 术语。

            ```bash
            nix flake init --template github:luochen1990/flake-fhs#std
            ```
          </TabItem>
          <TabItem label="最小化" icon="minimize">
            **特点**：仅包含 `flake.nix`，由你自己创建目录结构。

            ```bash
            nix flake init --template github:luochen1990/flake-fhs#zero
            ```
          </TabItem>
        </Tabs>
      </TabItem>

      <TabItem label="现有项目集成" icon="puzzle">
        适用于为**非纯 Nix 项目** (如 Python, Node.js, Rust) 添加 Nix 支持。

        它会为你生成最小化的 `flake.nix` 和必要的辅助文件，而不干扰现有的项目结构。

        ```bash
        # 将 Nix 支持集成到当前项目
        nix flake init --template github:luochen1990/flake-fhs#project
        ```
      </TabItem>
    </Tabs>

2.  **添加一个包**

    创建文件 `pkgs/hello.nix`：

    ```nix
    { writeShellScriptBin }:
    writeShellScriptBin "hello" ''
      echo "Hello, Flake FHS!"
    ''
    ```

    *无需手动修改 `flake.nix`，框架会自动发现它。*

3.  **构建与运行**

    ```bash
    nix build .#hello
    nix run .#hello
    ```

</Steps>
