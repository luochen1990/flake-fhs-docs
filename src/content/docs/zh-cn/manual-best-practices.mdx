以下是使用 Flake FHS 的推荐实践，遵循这些原则可以帮助你构建更易维护的项目。

## 项目组织

*   **遵循约定**
    尽量使用框架默认的目录结构，减少自定义配置。统一的目录结构让项目更易于理解和维护。

*   **模块化**
    将复杂的系统配置拆分为小的、可复用的模块 (`modules/`)。避免在一个巨大的 `configuration.nix` 中堆砌所有配置。

*   **利用封装**
    如果一个包需要多个辅助文件，请使用**子目录包**（即创建 `pkgs/<name>/package.nix`）。该目录下的其他 `.nix` 文件（如 `helper.nix`）不会被自动扫描为独立包，从而保持对外接口的整洁。

## 开发流程

*   **快速开始**
    总是使用模板 (`nix flake init --template ...`) 来初始化新项目或组件，这能确保你获得正确的样板代码。

*   **持续检查**
    养成运行 `nix flake check` 的习惯，配合 `checks/` 目录下的测试用例。这能尽早发现构建错误。

*   **格式化**
    使用 `nix fmt` 保持代码风格统一。一致的代码风格有助于多人协作。

## 迁移与集成

*   **渐进式采用 (Progressive Migration)**
    如果你已有一个庞大的 flake，不必一次性重构所有内容。你可以仅使用 `flake-fhs` 管理部分输出 (如 `packages`)，其余部分暂时保持原样, 以实现逐步迁移。

    ```nix
    {
      inputs = {
        nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
        flake-fhs.url = "github:luochen1990/flake-fhs";
      };

      outputs = inputs@{ self, nixpkgs, flake-fhs, ... }:
        let
          # 1. 创建 flake-fhs 实例，但不直接作为 outputs 返回
          fhs = flake-fhs.lib.mkFlake { inherit inputs; } { };
        in
        {
          # 2. 从 flake-fhs 中"摘取"你想迁移的部分 (例如 packages)
          # fhs.packages 已经处理好了所有 system 的遍历
          packages = fhs.packages;

          # 3. 其他部分保持原有的手动定义方式
          nixosConfigurations.my-machine = nixpkgs.lib.nixosSystem {
            # ... 现有的复杂配置 ...
          };
        };
    }
    ```

    这种方式让你可以：
    - 逐步迁移各个部分，无需一次性重构
    - 在迁移过程中继续正常开发
    - 根据需要选择迁移的优先级和节奏

    **迁移建议**：优先迁移频繁修改的内容（如 `pkgs/`、`shells/`），稳定的内容可以稍后迁移。
