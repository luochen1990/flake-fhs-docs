`modules/` 目录用于组织可复用的 NixOS 模块。系统将根据目录特征自动识别并分类为三种互斥的模块类型，无需手动维护 `module-list.nix`。

## 模块分类

框架将模块分为三种互斥的类型，按复杂度从低到高依次为：**Single File Module**、**Traditional Directory Module** 和 **Guarded Directory Module**。

### 1. Single File Module（单文件模块）

**识别标识**：独立的 `.nix` 文件

**特点**：
- 直接导出，无 enable 自动注入机制
- 最简单的模块形式

**适用场景**：简单配置

### 2. Traditional Directory Module（传统目录模块）

**识别标识**：目录包含 `default.nix`（且无 `options.nix`）

**特点**：
- 直接导出，无 enable 自动注入机制
- **与 nixpkgs 上游兼容**：使用标准的 `default.nix` 结构，可以直接贡献给 nixpkgs
- 不支持嵌套（子目录中即便存在 `default.nix` 也不会被框架当成独立模块）
- 模块边界清晰，自行管理 imports

**适用场景**：预备贡献给 **nixpkgs 上游**的模块、需要手动控制要导入哪些 nix 文件的复杂模块

**示例**：
```
modules/tweaks/
├── default.nix  # 主模块
└── utils.nix    # 被 default.nix 显式导入
```

<Aside type="caution" title="重要限制">
Traditional 模块**不支持嵌套**。子目录中即便存在 `default.nix` 也不会被框架当成独立模块。如果需要嵌套，请使用 Guarded 模块。
</Aside>

### 3. Guarded Directory Module（受保护的目录模块）

**识别标识**：目录包含 `options.nix`

**特点**：
- 自动生成 `enable` 选项
- 配置文件自动包裹在 `lib.mkIf enable { ... }` 中
- 支持嵌套（子模块会检查所有父级的 enable）

**适用场景**：需要定义开关选项，并希望将复杂配置拆分到多个小文件中进行维护的**复杂模块**

**示例**：
```
modules/services/web-server/
├── options.nix  # 定义选项
├── config.nix   # 主配置（自动包裹在 mkIf 中）
└── logging.nix  # 日志配置（也会被包裹）
```

## 冲突检测

<Aside type="danger" title="options.nix 与 default.nix 冲突">
同一目录下**不能**同时存在 `options.nix` 和 `default.nix`。这会导致构建错误，因为框架无法确定模块类型。

**错误示例**：
```
modules/myapp/
├── options.nix  ❌ 冲突
└── default.nix  ❌ 冲突
```

**解决方案**：
- 选择 A：重命名 `default.nix` 为 `config.nix`（变为 Guarded 模块）
- 选择 B：删除 `options.nix`（变为 Traditional 模块）
</Aside>

## 使用方法

### 使用模块

在 `hosts/my-machine/configuration.nix` 中：

```nix
{
  # Guarded 模块需要显式启用
  services.web-server.enable = true;
  services.web-server.port = 9000;
  
  # Traditional 模块和 Single File 模块自动生效（无需 enable）
  # 它们的配置已自动应用
}
```

### 定义 Single File 模块

最简单的模块形式，直接编写配置：

```nix
# modules/personal/config.nix
{ config, lib, ... }:
{
  # 直接定义配置，无 enable 自动注入机制
  users.users.alice = {
    isNormalUser = true;
    extraGroups = [ "wheel" ];
  };
}
```

### 定义 Traditional 模块

适用于需要完全控制导入逻辑的场景：

```nix
# modules/tweaks/default.nix
{ config, lib, ... }:
{
  # 直接定义配置，无 enable 自动注入机制
  environment.systemPackages = [ /* ... */ ];
  
  # 可以显式导入同目录下的辅助文件
  imports = [ ./utils.nix ];
}
```

### 定义 Guarded 模块

以 `modules/services/web-server` 为例：

**1. `options.nix`: 定义接口**

<Aside type="tip" title="严格模式">
框架要求显式定义完整的选项路径（例如 `options.services.web-server`），并会检查其是否匹配目录结构。
</Aside>

<Aside type="tip" title="自动 Enable">
框架会自动在模块路径下生成 `enable` 选项（如果未手动定义）。
</Aside>

```nix
{ lib, ... }:
{
  options.services.web-server = {
    # 需完整写出 options.services.web-server
    port = lib.mkOption {
      type = lib.types.port;
      default = 8080;
    };
  };
}
```

**2. `config.nix`: 实现逻辑**

配置文件会自动被包裹在 `lib.mkIf` 中，**无需手动添加**。

```nix
{ config, pkgs, ... }:
let
  cfg = config.services.web-server;
in
{
  # 无需手动写 config = lib.mkIf cfg.enable { ... }
  # 框架已自动包裹

  systemd.services.web-server = {
    description = "Simple Python Web Server";
    wantedBy = [ "multi-user.target" ];
    serviceConfig = {
      ExecStart = "${pkgs.python3}/bin/python -m http.server ${toString cfg.port}";
      Restart = "always";
    };
  };
}
```

### 定义 Traditional 模块

适用于需要完全控制导入逻辑的场景：

```nix
# modules/tweaks/default.nix
{ config, lib, ... }:
{
  # 直接定义配置，无 enable 自动注入机制
  environment.systemPackages = [ /* ... */ ];
  
  # 可以显式导入同目录下的辅助文件
  imports = [ ./utils.nix ];
}
```

### 嵌套 Guarded 模块

Guarded 模块支持嵌套，子模块会检查**所有父级的 enable**：

<FileTree>

- modules/
  - network/  <Badge text="network.enable" variant="default" size="small" />
    - options.nix
    - config.nix
    - services/
      - web/  <Badge text="network.enable && network.services.web.enable" variant="default" size="small" />
        - options.nix
        - config.nix

</FileTree>

生成的 `mkIf` 条件：
- `network/config.nix`: `lib.mkIf config.network.enable { ... }`
- `network/services/web/config.nix`: `lib.mkIf (config.network.enable && config.network.services.web.enable) { ... }`

## 设计决策

<Aside type="note" title="为什么 Traditional 模块不支持嵌套？">
Traditional 模块被设计为兼容传统的 Nix 目录模块，由用户手动控制导入，嵌套在这里本就无甚意义。

- **Traditional = 兼容传统**：手动控制导入，与 nixpkgs 上游一致
- **Guarded = 可嵌套的**：支持层级化的 enable 控制

如果需要嵌套结构，请使用 Guarded 模块。
</Aside>
