## Problem Background

Many developers worry that migrating existing large Nix Flake projects to Flake FHS requires a one-time large-scale refactoring, which may lead to:

- High risk: Too many changes at once, prone to errors
- Heavy workload: Need to pause normal development for migration
- Team resistance: Too much change, difficult for team to adapt

## Flake FHS Solution

Flake FHS was designed with gradual adoption in mind from the beginning, supporting three adoption modes:

### Mode 1: Full Adoption

All outputs are automatically generated by Flake FHS:

```nix
{
  inputs.nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
  inputs.flake-fhs.url = "github:luochen1990/flake-fhs";

  outputs = inputs@{ flake-fhs, ... }:
    flake-fhs.lib.mkFlake { inherit inputs; } {
      # All outputs automatically generated
    };
}
```

Suitable for:
- New projects
- Existing projects willing to do complete refactoring

### Mode 2: Hybrid Mode

Some outputs generated by framework, some manually defined:

```nix
{
  inputs.nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
  inputs.flake-fhs.url = "github:luochen1990/flake-fhs";

  outputs = inputs@{ flake-fhs, ... }:
    flake-fhs.lib.mkFlake { inherit inputs; } {
      # Automatically generated outputs
    }
    // {
      # Manually defined additional outputs
      overlays.default = final: prev: {
        customPackage = final.callPackage ./overlays/custom.nix { };
      };
      
      nixosModules.legacy = import ./legacy-module.nix;
    };
}
```

Suitable for:
- Projects in migration
- Outputs with special requirements that cannot be expressed by the framework

### Mode 3: Gradual Migration

Migrate in stages according to priority:

#### Stage 1: Migrate Package Definitions

```bash
# 1. Create directory structure
mkdir -p pkgs

# 2. Move package definitions to pkgs/
# Extract package definition from flake.nix to pkgs/hello.nix
mv legacy-packages/hello.nix pkgs/hello.nix
```

At this point you can:
- First migrate the simplest packages
- Keep other outputs unchanged
- Verify package builds correctly

#### Stage 2: Migrate Modules

```bash
# 1. Create modules directory
mkdir -p modules

# 2. Migrate NixOS modules
mv legacy-modules/tailscale.nix modules/tailscale/options.nix
touch modules/tailscale/config.nix  # Add configuration implementation
```

At this point you can:
- Modules can still be used independently
- Gradually add auto-discovery and enable options

#### Stage 3: Migrate Host Configurations

```bash
# 1. Create hosts directory
mkdir -p hosts

# 2. Migrate NixOS configurations
mv legacy-hosts/laptop hosts/laptop/configuration.nix
```

Finally complete the full migration.

## Migration Strategy Recommendations

### Priority Ordering

Migrate in the following priority order:

1. **High Priority**: Frequently modified content
   - Development environments (shells/)
   - Test checks (checks/)
   - Common packages (pkgs/)

2. **Medium Priority**: Relatively stable content
   - Applications (apps/)
   - NixOS modules (modules/)

3. **Low Priority**: Rarely changing configurations
   - NixOS hosts (hosts/)
   - Library functions (lib/)

### Parallel Work Strategy

Teams can use branch strategy for parallel migration:

```bash
# Each person responsible for migrating one directory
git checkout -b migrate/pkgs
git checkout -b migrate/modules
git checkout -b migrate/shells

# Merge after completion
git merge migrate/pkgs
git merge migrate/modules
git merge migrate/shells
```

### Verification Strategy

Verify at each stage:

```bash
# Stage 1 verification
nix build .#hello  # Verify package build
nix develop        # Verify development environment

# Stage 2 verification
nixos-rebuild build --flake .#laptop  # Verify configuration

# Stage 3 verification
nix flake check    # Comprehensive verification
```

## Practical Case

### Case: Migrating from Traditional Flake

**Before Migration** (traditional flake.nix):

```nix
{
  outputs = { self, nixpkgs }:
    let
      system = "x86_64-linux";
      pkgs = import nixpkgs { inherit system; };
    in
    {
      packages.${system}.hello = pkgs.writeShellScriptBin "hello" ''
        echo "Hello"
      '';
      
      devShells.${system}.default = pkgs.mkShell {
        packages = [ pkgs.git pkgs.nixpkgs-fmt ];
      };
    };
}
```

**After Migration** (gradual migration):

```nix
{
  inputs.flake-fhs.url = "github:luochen1990/flake-fhs";

  outputs = inputs@{ flake-fhs, nixpkgs, ... }:
    flake-fhs.lib.mkFlake { inherit inputs; } {
      # Framework automatically discovers pkgs/ and shells/
    };
}
```

```nix
# pkgs/hello.nix
{ writeShellScriptBin }:
writeShellScriptBin "hello" ''
  echo "Hello"
''
```

```nix
# shells/default.nix
{ mkShell, git, nixpkgs-fmt }:
mkShell {
  packages = [ git nixpkgs-fmt ];
}
```

## Advantages Summary

| Feature | Traditional Migration | Gradual Adoption |
|---------|----------------------|------------------|
| Risk | High (one-time big change) | Low (step-by-step verification) |
| Downtime | Long-term blocking | No downtime required |
| Team Adaptation | Need training before use | Can learn while migrating |
| Rollback Difficulty | Difficult (too many changes) | Easy (small changes each time) |
| Value Demonstration | Only after migration complete | Benefit at each stage |
