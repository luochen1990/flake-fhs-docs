**Flake FHS** introduces a standard **Filesystem Hierarchy Standard (FHS)** to directly map Flake Outputs to the filesystem. **As long as you see the directory structure, you know where the corresponding Output is.**

## Target Audience

Flake FHS is especially for you if:

- **You're new to Nix** and want to migrate to Flake but don't know where to start
- **You have extensive NixOS experience** and with a growing config repository, you have a strong desire to refactor it for better maintainability
- **You just want a turnkey Flake experience** without caring about Flake's design details
- **You want to learn Flake project organization best practices**, even if you don't end up using this framework

<Aside type="note">
If you already have a deep understanding of Flake and are completely satisfied with your current project structure, Flake FHS may not bring significant additional value.
</Aside>

## Why Flake FHS

Traditional Nix Flake development has the following core problems:

### Problem 1: Repeated Boilerplate Code

To add a package, the traditional approach requires a `flake.nix` like this:

```nix
{
  inputs.nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";

  outputs = { self, nixpkgs }:
    let
      systems = [ "x86_64-linux" "aarch64-linux" ];
      forAllSystems = f: nixpkgs.lib.genAttrs systems (system: f {
        pkgs = import nixpkgs { inherit system; };
      });
    in
    {
      packages = forAllSystems ({ pkgs }: {
        hello = pkgs.writeShellScriptBin "hello" ''
          echo "Hello from Flake FHS!"
        '';
      });
    };
}
```

**Issues**:
- The `forAllSystems` pattern is repeated in every project
- System traversal logic mixed with package definitions
- Adding new packages requires modifying `flake.nix`, error-prone

**Flake FHS Solution**:

`flake.nix` only needs this:

```nix
{
  inputs.nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";
  inputs.flake-fhs.url = "github:luochen1990/flake-fhs";

  outputs = inputs@{ flake-fhs, ... }:
    flake-fhs.lib.mkFlake { inherit inputs; } { };
}
```

To add a package, just create `pkgs/hello.nix`:

```nix
{ writeShellScriptBin }:
writeShellScriptBin "hello" ''
  echo "Hello from Flake FHS!"
''
```

The framework automatically traverses systems, discovers packages, injects dependencies, and generates outputs.

### Problem 2: Difficult Source Location

When taking over a new project, finding a package's source location often requires:
- Reading complex `flake.nix` files
- Understanding project-specific organization
- Each project has its own directory structure conventions

**Flake FHS Solution**:

Defines clear mappings from directories to flake outputs:

| Directory | Output | Nix Command |
|------|-------------|-----------|
| `pkgs/` | `packages.<system>.<name>` | `nix build .#<name>` |
| `apps/` | `apps.<system>.<name>` | `nix run .#<name>` |
| `hosts/` | `nixosConfigurations.<name>` | `nixos-rebuild switch --flake .#<name>` |
| `modules/` | `nixosModules.<name>` | - |
| `checks/` | `checks.<system>.<name>` | `nix flake check .#<name>` |
| `shells/` | `devShells.<system>.<name>` | `nix develop .#<name>` |
| `lib/` | `lib.<name>` | `nix eval .#lib.<name>` |
| `templates/` | `templates.<name>` | `nix flake init --template .#<name>` |

See the directory structure and immediately know where the output is. Code location is predictable.

### Problem 3: Tedious Module Splitting

Traditional NixOS module splitting requires:

```nix
{
  imports = [
    ./modules/tailscale.nix
    ./modules/ssh.nix
    ./modules/firewall.nix
  ];
}
```

**Issues**:
- Must manually maintain import lists
- Easy to miss a module
- Fine-grained splitting requires `lib.mkIf` boilerplate in every file

**Flake FHS Solution**:

Put modules in the `modules/` directory for auto-discovery and registration. Combined with **Guarded Module**, the `enable` option and conditional logic are also auto-generated.

### Problem 4: Difficult Sub-Scope Package Management

nixpkgs' `pkgs/by-name/` mechanism only handles packages in the top-level `pkgs` scope. For sub-scopes (like `python3Packages`, `haskellPackages`), you need to:
- Manually call `pkgs.python3Packages.callPackage`
- Repeat scope specification in each package
- Scattered dependency injection logic, hard to manage uniformly

**Flake FHS Solution: Scoped Package Tree**

Define sub-scopes via `scope.nix`, packages in subdirectories automatically inherit:

```
pkgs/
└── python/
    ├── scope.nix           # Define scope
    ├── my-app/
    │   └── package.nix     # Auto-inherits python3Packages scope
    └── my-lib/
        └── package.nix
```

### Problem 5: Lack of Unified Standards

Different Nix projects often adopt different directory structures, naming conventions, and organization methods, leading to:
- Team members need to learn new project conventions
- Difficult to develop universal automation tools
- Increased code review and understanding costs

**Flake FHS Solution**:

Provides standardized directory structure applicable to all projects:

```
flake.nix              # Framework config
pkgs/                  # Packages
apps/                  # Applications
hosts/                 # NixOS configurations
modules/               # NixOS modules
shells/                # Development shells
```

## Core Scenario Comparison

Now that we understand the problems and solutions, let's see Flake FHS's advantages through concrete scenarios.

### Scenario 1: Module Splitting

Split configuration into multiple modules for separation of concerns.

<Tabs>
<TabItem label="Traditional 😫">

**Simple Splitting**: Requires manually maintaining the `imports` list:

```nix
// configuration.nix
{
  imports = [
    ./modules/desktop.nix
    ./modules/network.nix
    ./modules/services.nix
    // Easy to forget adding new modules
  ];
}
```

**Fine-Grained Splitting**: Boilerplate required in every file:

```nix
// modules/my-website/options.nix
{ lib, ... }:
{
  options.my-website.enable = lib.mkEnableOption "my website";
}
```

```nix
// modules/my-website/nginx.nix
{ config, lib, ... }:
{
  config = lib.mkIf config.my-website.enable {
    services.nginx.enable = true;
  };
}
```

```nix
// modules/my-website/redis.nix
{ config, lib, ... }:
{
  config = lib.mkIf config.my-website.enable {
    services.redis.enable = true;
  };
}
```

```nix
// configuration.nix
{
  imports = [
    ./modules/my-website/options.nix
    ./modules/my-website/nginx.nix
    ./modules/my-website/redis.nix
    // Easy to forget adding new files
  ];
}
```

**Splitting has a cost**: Must manually define options, maintain imports, and repeat boilerplate.

</TabItem>

<TabItem label="Flake FHS ✨">

**Simple Splitting**: Just create directories, auto-discovered:

```
modules/
├── desktop/
│   ├── options.nix
│   └── config.nix
├── network/
│   ├── options.nix
│   └── config.nix
└── services/
    ├── options.nix
    └── config.nix
```

**Fine-Grained Splitting**: Guarded Module auto-wraps, no boilerplate needed:

```
modules/
└── my-website/
    ├── options.nix    # Can be empty, enable option auto-generated
    ├── nginx.nix      # Auto-wrapped with mkIf
    └── redis.nix      # Auto-wrapped with mkIf
```

```nix
// modules/my-website/nginx.nix
{ ... }:
{
  // Automatically wrapped as: lib.mkIf config.my-website.enable { ... }
  services.nginx.enable = true;
}
```

**Splitting is zero-cost**: Split freely, no boilerplate required.

</TabItem>
</Tabs>

### Scenario 2: Multi-Host Management

Manage multiple NixOS machines (laptop, server, vm, etc.) in one repository.

<Tabs>
<TabItem label="Traditional 😫">

Each host has its own Flake repository, no configuration sharing:

```
~/config-laptop/
├── flake.nix
└── configuration.nix

~/config-server/
├── flake.nix
└── configuration.nix
```

Must manually copy-paste shared configuration, hard to keep in sync.

</TabItem>

<TabItem label="Flake FHS ✨">

One repository for all hosts, shared modules:

```
my-nixos/
├── flake.nix
├── modules/
│   ├── tweaks.nix       # System optimizations
│   └── my-website.nix   # Website service
└── hosts/
    ├── laptop/
    │   └── configuration.nix
    └── server/
        └── configuration.nix
```

```nix
// hosts/laptop/configuration.nix
{ ... }:
{
  networking.hostName = "laptop";
  tweaks.enable = true;       // Only system optimizations
}
```

```nix
// hosts/server/configuration.nix
{ ... }:
{
  networking.hostName = "server";
  tweaks.enable = true;       // Shared module
  my-website.enable = true;   // Host-specific
}
```

</TabItem>
</Tabs>

## Comparison Summary

| Dimension | Traditional | Flake FHS |
|------|----------|-----------|
| Boilerplate Code | Repeatedly write `forAllSystems` etc. | Zero boilerplate |
| Source Location | Need to read `flake.nix` | Directory = mapping, clear at a glance |
| Module Splitting | Manually maintain imports + write `mkIf` in each file | Auto-discovery + Guarded Module |
| Sub-Scope Packages | Manually call `python3Packages.callPackage` | Scoped Package Tree |
| Project Standards | Each project has own conventions | Unified directory standard |
| Adding Content | Modify `flake.nix` | Just create a file |
| Learning Curve | Need deep understanding of Flake API | Just understand directory conventions |
