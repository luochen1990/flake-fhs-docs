The `modules/` directory is used to organize reusable NixOS modules. The system automatically identifies and classifies them into three mutually exclusive module types based on directory characteristics, without manually maintaining `module-list.nix`.

## Module Types

The framework divides modules into three mutually exclusive types, ordered from simplest to most complex: **Single File Module**, **Traditional Directory Module**, and **Guarded Directory Module**.

### 1. Single File Module

**Identifier**: Standalone `.nix` file

**Features**:
- Direct export, no auto-injected enable mechanism
- Simplest module form

**Use Case**: Simple configurations

### 2. Traditional Directory Module

**Identifier**: Directory contains `default.nix` (no `options.nix`)

**Features**:
- Direct export, no auto-injected enable mechanism
- **Compatible with nixpkgs upstream**: Uses standard `default.nix` structure, can be directly contributed to nixpkgs
- No nesting support (subdirectories with `default.nix` are NOT recognized by the framework as independent modules)
- Clear module boundaries, manages its own imports

**Use Case**: Modules **prepared for contributing to nixpkgs upstream**, complex modules requiring manual control over which nix files to import

**Example**:
```
modules/tweaks/
├── default.nix  # Main module
└── utils.nix    # Explicitly imported by default.nix
```

<Aside type="caution" title="Important Limitation">
Traditional modules **do NOT support nesting**. Subdirectories with `default.nix` will NOT be recognized by the framework as independent modules. Use Guarded modules if nesting is needed.
</Aside>

### 3. Guarded Directory Module

**Identifier**: Directory contains `options.nix`

**Features**:
- Auto-generates `enable` option
- Config files automatically wrapped in `lib.mkIf enable { ... }`
- Supports nesting (child modules check ALL parent enables)

**Use Case**: **Complex modules** that need an enable option and want to split complex configurations into multiple smaller files for maintainability

**Example**:
```
modules/services/web-server/
├── options.nix  # Define options
├── config.nix   # Main configuration (auto-wrapped in mkIf)
└── logging.nix  # Logging configuration (also wrapped)
```

## Conflict Detection

<Aside type="danger" title="options.nix and default.nix Conflict">
A directory **cannot** have both `options.nix` and `default.nix`. This causes a build error because the framework cannot determine the module type.

**Invalid Example**:
```
modules/myapp/
├── options.nix  ❌ Conflict
└── default.nix  ❌ Conflict
```

**Solutions**:
- Option A: Rename `default.nix` to `config.nix` (becomes Guarded module)
- Option B: Remove `options.nix` (becomes Traditional module)
</Aside>

## Usage

### Use Modules

In `hosts/my-machine/configuration.nix`:

```nix
{
  # Guarded modules need explicit enablement
  services.web-server.enable = true;
  services.web-server.port = 9000;
  
  # Traditional modules and Single File modules are auto-applied (no enable needed)
  # Their configurations are automatically in effect
}
```

### Define a Single File Module

The simplest module form, just write configuration directly:

```nix
// modules/personal/config.nix
{ config, lib, ... }:
{
  # Direct configuration, no auto-injected enable mechanism
  users.users.alice = {
    isNormalUser = true;
    extraGroups = [ "wheel" ];
  };
}
```

### Define a Traditional Module

For scenarios requiring full control over import logic:

```nix
// modules/tweaks/default.nix
{ config, lib, ... }:
{
  # Direct configuration, no auto-injected enable mechanism
  environment.systemPackages = [ /* ... */ ];
  
  # Can explicitly import helper files in the same directory
  imports = [ ./utils.nix ];
}
```

### Define a Guarded Module

Take `modules/services/web-server` as an example:

**1. `options.nix`: Define Interface**

<Aside type="tip" title="Strict Mode">
The framework requires explicitly defining the full option path (e.g., `options.services.web-server`), and checks if it matches the directory structure.
</Aside>

<Aside type="tip" title="Auto Enable">
The framework automatically generates an `enable` option under the module path (if not manually defined).
</Aside>

```nix
{ lib, ... }:
{
  options.services.web-server = {
    # Must write out full options.services.web-server
    port = lib.mkOption {
      type = lib.types.port;
      default = 8080;
    };
  };
}
```

**2. `config.nix`: Implementation Logic**

Config files are automatically wrapped in `lib.mkIf`, **no need to add manually**.

```nix
{ config, pkgs, ... }:
let
  cfg = config.services.web-server;
in
{
  # No need to manually write config = lib.mkIf cfg.enable { ... }
  # Framework auto-wraps this

  systemd.services.web-server = {
    description = "Simple Python Web Server";
    wantedBy = [ "multi-user.target" ];
    serviceConfig = {
      ExecStart = "${pkgs.python3}/bin/python -m http.server ${toString cfg.port}";
      Restart = "always";
    };
  };
}
```

### Define a Traditional Module

For scenarios requiring full control over import logic:

```nix
// modules/tweaks/default.nix
{ config, lib, ... }:
{
  # Direct configuration, no auto-injected enable mechanism
  environment.systemPackages = [ /* ... */ ];
  
  # Can explicitly import helper files in the same directory
  imports = [ ./utils.nix ];
}
```

### Nested Guarded Modules

Guarded modules support nesting. Child modules check **ALL parent enables**:

<FileTree>

- modules/
  - network/  <Badge text="network.enable" variant="default" size="small" />
    - options.nix
    - config.nix
    - services/
      - web/  <Badge text="network.enable && network.services.web.enable" variant="default" size="small" />
        - options.nix
        - config.nix

</FileTree>

Generated `mkIf` conditions:
- `network/config.nix`: `lib.mkIf config.network.enable { ... }`
- `network/services/web/config.nix`: `lib.mkIf (config.network.enable && config.network.services.web.enable) { ... }`

## Design Decisions

<Aside type="note" title="Why Don't Traditional Modules Support Nesting?">
Traditional modules are designed to be compatible with traditional Nix directory modules, with users manually controlling imports, so nesting has little meaning here.

- **Traditional = Compatible with tradition**: Manual import control, consistent with nixpkgs upstream
- **Guarded = Nestable**: Supports hierarchical enable control

Use Guarded modules if you need nesting structures.
</Aside>
